## 1. 产品概述

读书报告生成系统（ReadKing）是一个基于大模型技术的智能文档处理平台，专注于为用户提供高质量、个性化的读书报告生成服务。本项目旨在展示大模型应用开发（Agent/RAG）的核心技术能力，适用于求职大模型相关岗位。

系统将重点放在**后端大模型逻辑**和**Agent协作机制**上，前端保持极简且美观的设计风格。

## 2. 核心功能

### 2.1 功能重点 (面向面试)
*   **Agent 协作**: 使用 LangGraph 构建多 Agent 工作流。
*   **RAG 增强**: 针对长文本书籍的精准检索与分析。
*   **个性化输入**: 结合用户提供的“个人理解与感受”生成更具深度的报告。
*   **局部优化**: 基于反馈的迭代生成能力。

### 2.2 功能模块
1.  **书籍处理**: 支持上传电子书（PDF/TXT/EPUB），重点优化中文书籍解析。
2.  **全书智能扫描 (无输入处理)**:
    *   当用户未提供具体要求或感悟时，系统启动“分层摘要”模式。
    *   自动提取目录、章节核心，利用 Map-Reduce 架构生成全书全景画像。
3.  **报告生成预设**:
    *   书籍基础信息。
    *   **个人读后感/理解**（可选：允许用户输入自己的见解，Agent 将结合书籍内容进行深化）。
    *   字数、风格（学术、感性、犀利等）、自定义要求。
4.  **智能报告生成**:
    *   自动提取书籍核心观点。
    *   结合 RAG 检索深度细节。
    *   结合用户个人见解进行升华。
5.  **反馈与优化**: 支持用户对生成结果的特定段落提出修改意见。
6.  **导出下载**: 支持将最终报告导出为 Markdown 或 PDF。

## 3. 用户流程

1.  **上传与输入**: 上传书籍 -> 填写预设要求 -> **输入个人读后感/理解**。
2.  **AI 生成**: 后端 Agent 协作，显示生成进度。
3.  **预览与调整**: 查看报告 -> 选中段落反馈修改 -> AI 重新生成。
4.  **完成导出**: 下载报告到本地。

## 4. 用户界面 (UI) 设计

*   **设计原则**: 极简主义，突出内容，易于阅读。
*   **主色调**: 
    *   **浅棕色 (#D2B48C)**: 用于侧边栏或主要装饰。
    *   **奶油色 (#FFFDD0)**: 作为主要背景色，提供温暖的阅读感。
    *   **深褐色 (#5D4037)**: 用于正文字体，保证对比度。
*   **风格**: 类似 Notion 或现代博客的清爽布局。

## 5. 技术特色与成本优化

*   **低成本方案**: 
    *   **LLM (默认本地)**: 优先使用本地 **Ollama** 承载生成模型，推理成本≈0，便于离线演示与面试展示。
    *   **LLM (回退方案)**: 如果本机实测带不动或延迟不可接受，再切换到 **DeepSeek-V3/R1** 或 **SiliconFlow**（有免费额度/低单价）作为云端推理。
    *   **Embedding**: 使用本地开源模型 (如 `shibing624/text2vec-base-chinese`)，无需 API 调用费用。
    *   **存储**: 本地 SQLite + ChromaDB，零云端存储成本。
*   **一键切换**:
    *   通过环境变量选择 LLM Provider（`ollama`/`deepseek`/`siliconflow`），不改动 LangGraph 工作流与业务逻辑。
*   **全书扫描策略 (Map-Reduce)**:
    *   针对超长书籍，采用分块摘要再聚合的模式，避免一次性输入过长导致的费用暴涨或 Token 溢出。
*   **中文优化**: 专门针对中文书籍的 Token 限制和语义理解进行提示词优化。
*   **多 Agent 架构**: 
    *   `Reader Agent`: 负责精读、提取与 Map-Reduce 摘要。
    *   `Writer Agent`: 负责撰写。
    *   `Reviewer Agent`: 负责检查逻辑与风格。
*   **MCP 集成**: 实时获取书籍外部评价或作者背景。

## 6. MVP 与开发周期评估 (1人人力)

### 6.1 MVP (最小可行性产品) - 2-3 周
*   **目标**: 实现从书籍上传到报告生成的完整 AI 闭环。
*   **核心功能**:
    *   PDF/TXT 文件解析。
    *   基础 RAG (向量检索)。
    *   单 Agent 生成流程 (包含个人理解输入)。
    *   极简的前端页面 (FastAPI + Jinja2 或简单 React)。
*   **时间分配**:
    *   后端框架与数据模型 (3天)。
    *   RAG 与 Agent 逻辑开发 (7-10天)。
    *   前端页面与集成 (3-4天)。
    *   调试与优化 (2天)。

### 6.2 完整版 (面试展示级) - 6-8 周
*   **目标**: 具备高鲁棒性、多 Agent 复杂协作和精致 UI。
*   **增加功能**:
    *   多 Agent 协作流 (LangGraph)。
    *   局部优化反馈回路。
    *   MCP 外部工具集成。
    *   异步任务处理与进度实时推送。
    *   精致的响应式 UI 设计。
*   **时间分配**:
    *   LangGraph 复杂工作流设计 (10天)。
    *   MCP 与外部工具集成 (5天)。
    *   局部优化算法优化 (7天)。
    *   高级 UI 与动效实现 (7天)。
    *   性能压力测试与 Prompt 工程调优 (7天)。
